# sage-wiki

Реализация [идеи Андрея Карпати](https://x.com/karpathy/status/2039805659525644595) — персональная база знаний, компилируемая с помощью LLM. Разработано с использованием [Sage Framework](https://github.com/xoai/sage).

Некоторые уроки, извлечённые при создании sage-wiki, описаны [здесь](https://x.com/xoai/status/2040936964799795503).

Загрузите свои статьи, научные работы и заметки. sage-wiki компилирует их в структурированную, взаимосвязанную вики — с извлечёнными концепциями, обнаруженными перекрёстными ссылками и полнотекстовым поиском.

- **Источники на входе — вики на выходе.** Добавьте документы в папку. LLM прочитает, обобщит, извлечёт концепции и создаст взаимосвязанные статьи.
- **Масштабируется до 100K+ документов.** Многоуровневая компиляция индексирует всё быстро, компилирует только важное. Хранилище на 100K документов становится доступным для поиска за часы, а не за месяцы.
- **Накопление знаний.** Каждый новый источник обогащает существующие статьи. Вики становится умнее по мере роста.
- **Работает с вашими инструментами.** Нативно открывается в Obsidian. Подключается к любому LLM-агенту через MCP. Работает как единый бинарный файл — совместим с API-ключами или вашей подпиской на LLM.
- **Задавайте вопросы своей вики.** Улучшенный поиск с индексацией на уровне фрагментов, расширением запросов через LLM и переранжированием. Задавайте вопросы на естественном языке и получайте ответы с цитированием источников.
- **Компиляция по запросу.** Агенты могут инициировать компиляцию по конкретным темам через MCP. Результаты поиска сигнализируют о наличии нескомпилированных источников.

https://github.com/user-attachments/assets/c35ee202-e9df-4ccd-b520-8f057163ff26

_Точки на внешней границе представляют резюме всех документов в базе знаний, а точки во внутреннем круге — концепции, извлечённые из базы знаний, со связями, показывающими, как эти концепции соединены друг с другом._

## Установка

```bash
# Только CLI (без веб-интерфейса)
go install github.com/xoai/sage-wiki/cmd/sage-wiki@latest

# С веб-интерфейсом (требуется Node.js для сборки фронтенда)
git clone https://github.com/xoai/sage-wiki.git && cd sage-wiki
cd web && npm install && npm run build && cd ..
go build -tags webui -o sage-wiki ./cmd/sage-wiki/
```

## Поддерживаемые форматы источников

| Формат      | Расширения                              | Что извлекается                                             |
| ----------- | --------------------------------------- | ----------------------------------------------------------- |
| Markdown    | `.md`                                   | Основной текст с отдельным разбором frontmatter             |
| PDF         | `.pdf`                                  | Полный текст через чистый Go-экстрактор                     |
| Word        | `.docx`                                 | Текст документа из XML                                      |
| Excel       | `.xlsx`                                 | Значения ячеек и данные листов                              |
| PowerPoint  | `.pptx`                                 | Текстовое содержимое слайдов                                |
| CSV         | `.csv`                                  | Заголовки + строки (до 1000 строк)                          |
| EPUB        | `.epub`                                 | Текст глав из XHTML                                         |
| Email       | `.eml`                                  | Заголовки (от/кому/тема/дата) + тело письма                 |
| Текст       | `.txt`, `.log`                          | Исходное содержимое                                         |
| Транскрипции| `.vtt`, `.srt`                          | Исходное содержимое                                         |
| Изображения | `.png`, `.jpg`, `.gif`, `.webp`, `.svg` | Описание через LLM с поддержкой зрения (подпись, содержание, видимый текст) |
| Код         | `.go`, `.py`, `.js`, `.ts`, `.rs` и др. | Исходный код                                                |

Просто поместите файлы в папку источников — sage-wiki автоматически определит формат. Для изображений требуется LLM с поддержкой зрения (Gemini, Claude, GPT-4o).

Нужен формат, которого нет в списке? sage-wiki поддерживает **внешние парсеры** — скрипты на любом языке, которые читают stdin и выводят текст в stdout. См. [Внешние парсеры](#внешние-парсеры) ниже.

## Быстрый старт

![Конвейер компилятора](sage-wiki-compiler-pipeline.png)

### С нуля (новый проект)

```bash
mkdir my-wiki && cd my-wiki
sage-wiki init
# Добавьте источники в raw/
cp ~/papers/*.pdf raw/papers/
cp ~/articles/*.md raw/articles/
# Отредактируйте config.yaml — добавьте API-ключ и выберите LLM
# Первая компиляция
sage-wiki compile
# Поиск
sage-wiki search "attention mechanism"
# Задать вопросы
sage-wiki query "How does flash attention optimize memory?"
# Интерактивная терминальная панель
sage-wiki tui
# Просмотр в браузере (требуется сборка с -tags webui)
sage-wiki serve --ui
# Отслеживание папки
sage-wiki compile --watch
```

### Оверлей хранилища (существующее хранилище Obsidian)

```bash
cd ~/Documents/MyVault
sage-wiki init --vault
# Отредактируйте config.yaml — укажите папки источников/игнорирования, добавьте API-ключ, выберите LLM
# Первая компиляция
sage-wiki compile
# Отслеживание хранилища
sage-wiki compile --watch
```

### Docker

```bash
# Загрузка из GitHub Container Registry
docker pull ghcr.io/xoai/sage-wiki:latest

# Или из Docker Hub
docker pull xoai/sage-wiki:latest

# Запуск с подключённым каталогом вики
docker run -d -p 3333:3333 -v ./my-wiki:/wiki -e GEMINI_API_KEY=... ghcr.io/xoai/sage-wiki

# Или сборка из исходного кода
docker build -t sage-wiki .
docker run -d -p 3333:3333 -v ./my-wiki:/wiki -e GEMINI_API_KEY=... sage-wiki
```

Доступные теги: `:latest` (основная ветка), `:v1.0.0` (релизы), `:sha-abc1234` (конкретные коммиты). Мультиархитектура: `linux/amd64` и `linux/arm64`.

Смотрите [руководство по self-hosting](docs/guides/self-hosted-server.md) для Docker Compose, синхронизации через Syncthing, обратного прокси и настройки LLM-провайдера.

## TUI

```bash
sage-wiki tui
```

Полнофункциональная терминальная панель с 4 вкладками:

- **[F1] Обзор** — Навигация по статьям по разделам (концепции, резюме, выходные данные). Стрелками выберите, Enter для чтения с отрисовкой markdown через glamour, Esc для возврата.
- **[F2] Поиск** — Нечёткий поиск с разделённой панелью предварительного просмотра. Печатайте для фильтрации, результаты ранжируются по гибридному баллу, Enter для открытия в `$EDITOR`.
- **[F3] Вопрос-ответ** — Потоковый диалоговый Q&A. Задавайте вопросы, получайте синтезированные LLM ответы с цитированием источников. Ctrl+S сохраняет ответ в outputs/.
- **[F4] Компиляция** — Панель компиляции в реальном времени. Отслеживает каталоги источников на предмет изменений и автоматически перекомпилирует. Просмотр скомпилированных файлов с предварительным просмотром.

Переключение вкладок: `F1`-`F4` из любой вкладки, `1`-`4` в Обзоре/Компиляции, `Esc` возвращает в Обзор. Выход — `Ctrl+C`.

## Веб-интерфейс

![Веб-интерфейс Sage-Wiki](sage-wiki-webui.png)

sage-wiki включает опциональный браузерный просмотрщик для чтения и исследования вашей вики.

```bash
sage-wiki serve --ui
# Открывается по адресу http://127.0.0.1:3333
```

Возможности:

- **Браузер статей** с отрисовкой markdown, подсветкой синтаксиса и кликабельными `[[вики-ссылками]]`
- **Гибридный поиск** с ранжированными результатами и фрагментами
- **Граф знаний** — интерактивная визуализация концепций и их связей на основе силового графа
- **Потоковый Q&A** — задавайте вопросы и получайте синтезированные LLM ответы с цитированием источников
- **Оглавление** со scroll-spy или переключение на вид графа
- **Тёмная/светлая тема** с определением системных предпочтений
- **Обнаружение битых ссылок** — отсутствующие ссылки на статьи отображаются серым цветом

Веб-интерфейс построен на Preact + Tailwind CSS и встроен в Go-бинарник через `go:embed`. Добавляет ~1.2 МБ (gzip) к размеру бинарника. Для сборки без веб-интерфейса опустите флаг `-tags webui` — бинарник по-прежнему будет работать для всех CLI- и MCP-операций.

Параметры:

- `--port 3333` — изменить порт (по умолчанию 3333)
- `--bind 0.0.0.0` — открыть доступ в сети (по умолчанию только localhost, без аутентификации)

## Конфигурация

`config.yaml` создаётся командой `sage-wiki init`. Полный пример:

```yaml
version: 1
project: my-research
description: "Personal research wiki"

# Папки источников для отслеживания и компиляции
sources:
  - path: raw # или папки хранилища, например Clippings/, Papers/
    type: auto # автоопределение по расширению файла
    watch: true

output: wiki # каталог скомпилированных данных (_wiki для оверлея хранилища)

# Папки, которые никогда не читаются и не отправляются в API (режим оверлея хранилища)
# ignore:
#   - Daily Notes
#   - Personal

# Провайдер LLM
# Поддерживаются: anthropic, openai, gemini, ollama, openai-compatible, qwen
# Для OpenRouter или других OpenAI-совместимых провайдеров:
#   provider: openai-compatible
#   base_url: https://openrouter.ai/api/v1
# Для Alibaba Cloud DashScope Qwen:
#   provider: qwen
#   api_key: ${DASHSCOPE_API_KEY}
api:
  provider: gemini
  api_key: ${GEMINI_API_KEY} # поддерживается подстановка переменных окружения
  # auth: subscription          # использовать учётные данные подписки вместо api_key
                                # требуется: sage-wiki auth login --provider <name>
                                # поддерживаемые провайдеры: openai, anthropic, gemini
  # base_url:                   # пользовательская точка доступа (OpenRouter, Azure и т.д.)
  # rate_limit: 60              # запросов в минуту
  # extra_params:               # параметры, специфичные для провайдера, добавляемые в тело запроса
  #   enable_thinking: false    # например, отключить режим «мышления» Qwen
  #   reasoning_effort: low     # например, управление рассуждениями DeepSeek

# Модель для каждой задачи — дешёвые модели для массовых операций, качественные для написания
models:
  summarize: gemini-3-flash-preview
  extract: gemini-3-flash-preview
  write: gemini-3-flash-preview
  lint: gemini-3-flash-preview
  query: gemini-3-flash-preview

# Провайдер эмбеддингов (опционально — автоопределение из провайдера api)
# Переопределите для использования другого провайдера для эмбеддингов
embed:
  provider: auto # auto, openai, gemini, ollama, voyage, mistral
  # model: text-embedding-3-small
  # api_key: ${OPENAI_API_KEY}  # отдельный ключ для эмбеддингов
  # base_url:                   # отдельная точка доступа
  # rate_limit: 0              # ограничение RPM для эмбеддингов (0 = без лимита; установите 1200 для Gemini Tier 1)

# Примечание о мультипровайдерности:
# Секция api настраивает основной LLM-провайдер для всех задач компилятора
# и запросов (summarize, extract, write, lint, query). Секция embed
# может использовать ДРУГОЙ провайдер для эмбеддингов — со своими api_key, base_url
# и rate_limit. Это позволяет комбинировать провайдеров для оптимизации стоимости или качества:
#
#   api:
#     provider: anthropic                    # Claude для генерации
#     api_key: ${ANTHROPIC_API_KEY}
#   models:
#     summarize: claude-haiku-4-5-20251001   # дешёвая модель для массовой работы
#     write: claude-sonnet-4-20250514        # качественная модель для статей
#     query: claude-sonnet-4-20250514
#   embed:
#     provider: openai                       # OpenAI для эмбеддингов
#     model: text-embedding-3-small
#     api_key: ${OPENAI_API_KEY}
#
# С авторизацией по подписке вы можете аутентифицироваться у нескольких провайдеров:
#   sage-wiki auth login --provider anthropic
#   sage-wiki auth import --provider gemini
# Затем используйте Anthropic для генерации и Gemini для эмбеддингов.

compiler:
  max_parallel: 20 # параллельных LLM-вызовов (с адаптивным обратным давлением)
  debounce_seconds: 2 # задержка при режиме отслеживания
  summary_max_tokens: 2000
  article_max_tokens: 4000
  # extract_batch_size: 20     # резюме на вызов извлечения концепций (уменьшите, чтобы избежать обрезки JSON на больших корпусах)
  # extract_max_tokens: 8192   # максимум токенов на выходе для извлечения концепций (увеличьте до 16384, если извлечение обрезается)
  auto_commit: true # git commit после компиляции
  auto_lint: true # запуск линтера после компиляции
  mode: auto # standard, batch или auto (auto = batch при 10+ источниках)
  # estimate_before: false    # показать оценку стоимости перед компиляцией
  # prompt_cache: true        # включить кэширование промптов (по умолчанию: true)
  # batch_threshold: 10       # минимум источников для режима auto-batch
  # token_price_per_million: 0  # переопределить ценообразование (0 = использовать встроенное)
  # timezone: Asia/Shanghai   # часовой пояс IANA для пользовательских меток времени (по умолчанию: UTC)
  # article_fields:           # пользовательские поля frontmatter, извлекаемые из ответа LLM
  #   - language
  #   - domain

  # Многоуровневая компиляция — быстрая индексация, компиляция важного
  default_tier: 3 # 0=индекс, 1=индекс+эмбеддинг, 3=полная компиляция
  # tier_defaults:             # переопределение уровня по расширению
  #   json: 0                  # структурированные данные — только индекс
  #   yaml: 0
  #   lock: 0
  #   md: 1                    # проза — индекс + эмбеддинг
  #   go: 1                    # код — индекс + эмбеддинг + разбор
  # auto_promote: true         # повышать до уровня 3 на основе попаданий при поиске
  # auto_demote: true          # понижать устаревшие статьи
  # split_threshold: 15000     # символов — разбивать большие документы для ускорения записи
  # dedup_threshold: 0.85      # косинусное сходство для дедупликации концепций
  # backpressure: true         # адаптивная параллельность при ограничениях скорости

search:
  hybrid_weight_bm25: 0.7 # вес BM25 относительно вектора
  hybrid_weight_vector: 0.3
  default_limit: 10
  # query_expansion: true     # расширение запросов через LLM для Q&A (по умолчанию: true)
  # rerank: true              # переранжирование через LLM для Q&A (по умолчанию: true)
  # chunk_size: 800           # токенов на фрагмент для индексации (100-5000)
  # graph_expansion: true     # расширение контекста через граф для Q&A (по умолчанию: true)
```

### Мультипровайдерная настройка

sage-wiki позволяет использовать различных LLM-провайдеров для разных задач. Секция `api` задаёт основного провайдера для генерации (summarize, extract, write, lint, query), а `embed` может использовать совершенно другого провайдера для эмбеддингов — со своими учётными данными и лимитами скорости.

**Варианты использования:**
- **Оптимизация стоимости** — дешёвая модель для массового реферирования, качественная модель для написания статей
- **Лучшее из каждой платформы** — Claude для генерации, OpenAI для эмбеддингов, Ollama для локального поиска
- **Комбинирование подписок** — используйте вашу подписку ChatGPT для генерации и подписку Gemini для эмбеддингов

**Пример: Claude для генерации + эмбеддинги OpenAI**

```yaml
api:
  provider: anthropic
  api_key: ${ANTHROPIC_API_KEY}

models:
  summarize: claude-haiku-4-5-20251001    # дешёвая модель для массовой работы
  extract: claude-haiku-4-5-20251001
  write: claude-sonnet-4-20250514         # качественная модель для статей
  lint: claude-haiku-4-5-20251001
  query: claude-sonnet-4-20250514

embed:
  provider: openai
  model: text-embedding-3-small
  api_key: ${OPENAI_API_KEY}
```

**Пример: Авторизация по подписке с двумя провайдерами**

```bash
sage-wiki auth login --provider anthropic
sage-wiki auth import --provider gemini
```

```yaml
api:
  provider: anthropic
  auth: subscription

embed:
  provider: gemini
  # api_key не нужен — используются импортированные учётные данные подписки Gemini
```

Секция `models` управляет тем, какая модель используется для каждой задачи, все в рамках основного провайдера. Разные модели могут иметь совершенно разное соотношение стоимости и качества — используйте меньшие модели (haiku, flash, mini) для массовых проходов, таких как реферирование, и более крупные модели (sonnet, pro) для написания статей и Q&A.

### Настраиваемые отношения

Онтология имеет 8 встроенных типов отношений: `implements`, `extends`, `optimizes`, `contradicts`, `cites`, `prerequisite_of`, `trades_off`, `derived_from`. Каждый имеет набор ключевых слов-синонимов по умолчанию, используемых для автоматического извлечения.

Вы можете настроить отношения через `ontology.relations` в `config.yaml`:

- **Расширить встроенный тип** — добавить синонимы (например, многоязычные ключевые слова) к существующему типу. Синонимы по умолчанию сохраняются; ваши добавляются к ним.
- **Добавить пользовательский тип** — определить новое имя отношения с его ключевыми словами-синонимами. Имена отношений должны быть в нижнем регистре `[a-z][a-z0-9_]*`.

Отношения извлекаются с помощью контекстной близости на уровне блоков — ключевое слово должно встречаться совместно с `[[вики-ссылкой]]` в одном абзаце или заголовочном блоке. Это предотвращает ложные связи от совпадений между абзацами.

Вы также можете ограничить, какие типы сущностей соединяет отношение:

```yaml
ontology:
  relation_types:
    - name: curated_by
      synonyms: ["curated by", "organized by"]
      valid_sources: [exhibition, program]
      valid_targets: [artist]
```

Когда указаны `valid_sources`/`valid_targets`, связи создаются только если тип сущности источника/цели совпадает. Пустое значение = все типы допустимы (по умолчанию).

Нулевая конфигурация = поведение идентично текущему. Существующие базы данных мигрируются автоматически при первом открытии. Смотрите [полное руководство](docs/guides/configurable-relations.md) для примеров, специфичных для предметной области, отношений с ограничением типов и описания работы извлечения.

## Оптимизация стоимости

sage-wiki отслеживает использование токенов и оценивает стоимость каждой компиляции. Три стратегии снижения затрат:

**Кэширование промптов** (по умолчанию: включено) — Повторное использование системных промптов в LLM-вызовах в рамках прохода компиляции. Anthropic и Gemini кэшируют явно; OpenAI кэширует автоматически. Экономия 50-90% на входных токенах.

**Batch API** — Отправка всех источников единым асинхронным пакетом со скидкой 50%. Доступно для Anthropic и OpenAI.

```bash
sage-wiki compile --batch       # отправить пакет, сохранить контрольную точку, выйти
sage-wiki compile               # проверить статус, получить результаты по готовности
```

**Оценка стоимости** — Предварительный просмотр стоимости перед началом:

```bash
sage-wiki compile --estimate    # показать разбивку стоимости, выйти
```

Или установите `compiler.estimate_before: true` в конфигурации для запроса подтверждения каждый раз.

**Автоматический режим** — Установите `compiler.mode: auto` и `compiler.batch_threshold: 10` для автоматического использования batch-режима при компиляции 10+ источников.

## Авторизация по подписке

Используйте существующую подписку на LLM вместо API-ключей. Поддерживаются ChatGPT Plus/Pro, Claude Pro/Max, GitHub Copilot и Google Gemini.

```bash
# Вход через браузер (OpenAI или Anthropic)
sage-wiki auth login --provider openai

# Или импорт из существующего CLI-инструмента
sage-wiki auth import --provider claude
sage-wiki auth import --provider copilot
sage-wiki auth import --provider gemini
```

Затем установите `api.auth: subscription` в вашем `config.yaml`:

```yaml
api:
  provider: openai
  auth: subscription
```

Все команды будут использовать учётные данные вашей подписки. Токены обновляются автоматически. Если токен истекает и не может быть обновлён, sage-wiki переключается на `api_key` с предупреждением.

**Ограничения:** Batch-режим недоступен при авторизации по подписке (автоматически отключается). Некоторые модели могут быть недоступны через токены подписки. Подробности в [руководстве по авторизации подписки](docs/guides/subscription-auth.md).

## Доверие к выходным данным

Когда sage-wiki отвечает на вопрос, ответ — это утверждение, сгенерированное LLM, а не проверенный факт. Без защитных механизмов ошибочные ответы индексируются в вики и загрязняют будущие запросы. Система доверия к выходным данным помещает новые ответы в карантин и требует верификации перед включением в поисковый корпус.

```yaml
# config.yaml
trust:
  include_outputs: verified  # "false" (исключить все), "verified" (только подтверждённые), "true" (устаревший)
  consensus_threshold: 3     # подтверждений для автоматического повышения
  grounding_threshold: 0.8   # минимальный балл обоснованности
  similarity_threshold: 0.85 # косинусное сходство для сопоставления вопросов
  auto_promote: true          # автоматическое повышение при выполнении порогов
```

**Как это работает:**

1. **Запрос** — sage-wiki отвечает на ваш вопрос. Ответ записывается в `wiki/under_review/` как ожидающий.
2. **Консенсус** — Если тот же вопрос задаётся повторно и даёт тот же ответ из других фрагментов источников, подтверждения накапливаются. Независимость оценивается по расстоянию Жаккара между наборами фрагментов.
3. **Обоснованность** — Запустите `sage-wiki verify` для проверки утверждений на соответствие исходным текстам через LLM-анализ следования.
4. **Повышение** — Когда оба порога (консенсус и обоснованность) достигнуты, ответ перемещается в `wiki/outputs/` и индексируется для поиска.

```bash
# Проверить ожидающие выходные данные
sage-wiki outputs list

# Запустить верификацию обоснованности
sage-wiki verify --all

# Вручную повысить доверенные выходные данные
sage-wiki outputs promote 2026-05-09-what-is-attention.md

# Разрешить конфликт (подтвердить один, отклонить остальные)
sage-wiki outputs resolve 2026-05-09-what-is-attention.md

# Очистить старые ожидающие выходные данные
sage-wiki outputs clean --older-than 90d

# Мигрировать существующие выходные данные в систему доверия
sage-wiki outputs migrate
```

Изменения источников во время `sage-wiki compile` автоматически понижают подтверждённые ответы, когда их цитируемые источники были изменены. Смотрите [руководство по доверию к выходным данным](docs/guides/output-trust.md) для полной архитектуры, справочника конфигурации и устранения проблем.

## Масштабирование для больших хранилищ

sage-wiki использует **многоуровневую компиляцию** для обработки хранилищ с 10K-100K+ документами. Вместо того чтобы пропускать всё через полный LLM-конвейер, источники маршрутизируются по уровням на основе типа файла и использования:

| Уровень | Что происходит | Стоимость | Время на документ |
|---------|---------------|-----------|-------------------|
| **0** — Только индекс | Полнотекстовый поиск FTS5 | Бесплатно | ~5 мс |
| **1** — Индекс + эмбеддинг | FTS5 + векторный эмбеддинг | ~$0.00002 | ~200 мс |
| **2** — Разбор кода | Структурное резюме через regex-парсер (без LLM) | Бесплатно | ~10 мс |
| **3** — Полная компиляция | Реферирование + извлечение концепций + написание статей | ~$0.05-0.15 | ~5-8 мин |

По умолчанию (`default_tier: 3`) все источники проходят полный LLM-конвейер — такое же поведение, как и до многоуровневой компиляции. Для больших хранилищ (10K+) установите `default_tier: 1`, чтобы проиндексировать всё за ~5.5 часов, а затем компилировать по запросу — когда агент запрашивает тему, поиск сигнализирует о нескомпилированных источниках, и `wiki_compile_topic` компилирует только этот кластер (~2 мин для 20 источников).

**Ключевые возможности:**
- **Настройки по типу файла** — JSON, YAML и lock-файлы автоматически переходят на Уровень 0. Настройка по расширению через `tier_defaults`.
- **Автоповышение** — Источники повышаются до Уровня 3 после 3+ попаданий при поиске или когда тематический кластер достигает 5+ источников.
- **Автопонижение** — Устаревшие статьи (90 дней без запросов) понижаются до Уровня 1 для перекомпиляции при следующем обращении.
- **Адаптивное обратное давление** — Параллельность автоматически подстраивается под ограничения скорости провайдера. Начинается с 20 параллельных вызовов, уменьшается вдвое при ошибках 429, восстанавливается автоматически.
- **10 парсеров кода** — Go (через go/ast), TypeScript, JavaScript, Python, Rust, Java, C, C++, Ruby, плюс извлечение ключей из JSON/YAML/TOML. Код получает структурные резюме без LLM-вызовов.
- **Компиляция по запросу** — `wiki_compile_topic("flash attention")` через MCP компилирует релевантные источники в реальном времени.
- **Оценка качества** — Автоматическое отслеживание покрытия источников, полноты извлечения и плотности перекрёстных ссылок для каждой статьи.

Смотрите [полное руководство по масштабированию](docs/guides/large-vault-performance.md) для конфигурации, примеров переопределения уровней и целевых показателей производительности.

## Качество поиска

sage-wiki использует улучшенный конвейер поиска для Q&A-запросов, вдохновлённый анализом подхода к извлечению [qmd](https://github.com/dmayboroda/qmd):

- **Индексация на уровне фрагментов** — Статьи разбиваются на фрагменты по ~800 токенов, каждый со своей записью FTS5 и векторным эмбеддингом. Поиск по «flash attention» находит релевантный абзац внутри статьи о Transformer на 3000 токенов.
- **Расширение запросов через LLM** — Один LLM-вызов генерирует переформулировки ключевых слов (для BM25), семантические переформулировки (для векторного поиска) и гипотетический ответ (для сходства эмбеддингов). Проверка сильного сигнала пропускает расширение, если топ-результат BM25 уже достаточно уверен.
- **Переранжирование через LLM** — Топ-15 кандидатов оцениваются LLM на релевантность. Позиционно-зависимое смешивание защищает высокоуверенные результаты поиска (ранги 1-3 получают 75% веса поиска, ранги 11+ получают 60% веса переранжирования).
- **Межъязыковой векторный поиск** — Полный перебор косинусного сходства по всем векторам фрагментов в сочетании с BM25 через RRF-слияние. Это гарантирует, что многоязычные запросы (например, запрос на польском к английскому контенту) находят семантически релевантные результаты даже при нулевом лексическом пересечении.
- **Расширение контекста через граф** — После извлечения 4-сигнальный графовый скорер находит связанные статьи через онтологию: прямые связи (×3.0), общие исходные документы (×4.0), общие соседи через Adamic-Adar (×1.5) и родство типов сущностей (×1.0). Это выявляет статьи, структурно связанные, но не найденные через ключевой/векторный поиск.
- **Контроль бюджета токенов** — Контекст запроса ограничен настраиваемым лимитом токенов (по умолчанию 8000), с ограничением на статью в 4000 токенов. Жадное заполнение отдаёт приоритет статьям с наивысшим баллом.

|                   | sage-wiki                                    | qmd               |
| ----------------- | -------------------------------------------- | ----------------- |
| Поиск фрагментов  | FTS5 + вектор (двухканальный)               | Только вектор     |
| Расширение запроса | На основе LLM (lex/vec/hyde)                | На основе LLM     |
| Переранжирование   | LLM + позиционно-зависимое смешивание       | Cross-encoder     |
| Графовый контекст  | 4-сигнальное расширение графа + 1-hop обход | Нет графа         |
| Стоимость запроса  | Бесплатно (Ollama) / ~$0.0006 (облако)      | Бесплатно (GGUF)  |

Нулевая конфигурация = все функции включены. С Ollama или другими локальными моделями улучшенный поиск полностью бесплатен — переранжирование автоматически отключается (локальные модели плохо справляются со структурированной JSON-оценкой), но поиск на уровне фрагментов и расширение запросов по-прежнему работают. С облачными LLM дополнительная стоимость незначительна (~$0.0006/запрос). Как расширение, так и переранжирование можно переключить через конфигурацию. Смотрите [полное руководство по качеству поиска](docs/guides/search-quality.md) для конфигурации, разбивки стоимости и детального сравнения.

## Настройка промптов

sage-wiki использует встроенные промпты для реферирования и написания статей. Для настройки:

```bash
sage-wiki init --prompts    # создаёт каталог prompts/ с настройками по умолчанию
```

Это создаёт редактируемые markdown-файлы:

```
prompts/
├── summarize-article.md    # как реферируются статьи
├── summarize-paper.md      # как реферируются научные работы
├── write-article.md        # как пишутся концептуальные статьи
├── extract-concepts.md     # как определяются концепции
└── caption-image.md        # как описываются изображения
```

Отредактируйте любой файл, чтобы изменить способ обработки этого типа. Добавляйте новые типы источников, создавая `summarize-{type}.md` (например, `summarize-dataset.md`). Удалите файл, чтобы вернуться к встроенному значению по умолчанию.

### Пользовательские поля frontmatter

Frontmatter статей формируется из двух источников: **данные основной истины** (название концепции, псевдонимы, источники, метка времени) всегда генерируются кодом, а **семантические поля** оцениваются LLM.

По умолчанию `confidence` — единственное поле, оцениваемое LLM. Чтобы добавить пользовательские поля:

1. Объявите их в `config.yaml`:

```yaml
compiler:
  article_fields:
    - language
    - domain
```

2. Обновите ваш шаблон `prompts/write-article.md`, чтобы запросить у LLM эти поля:

```
At the end of your response, state:
Language: (the primary language of the concept)
Domain: (the academic field, e.g., machine learning, biology)
Confidence: high, medium, or low
```

Ответы LLM извлекаются из тела статьи и автоматически объединяются в YAML frontmatter. Результирующий frontmatter выглядит так:

```yaml
---
concept: self-attention
aliases: ["scaled dot-product attention"]
sources: ["raw/transformer-paper.md"]
confidence: high
language: English
domain: machine learning
created_at: 2026-04-10T08:00:00+08:00
---
```

Поля основной истины (`concept`, `aliases`, `sources`, `created_at`) всегда точны — они берутся из прохода извлечения, а не из LLM. Семантические поля (`confidence` + ваши пользовательские поля) отражают оценку LLM.

## Пакеты расширения

Пакеты — это устанавливаемые профили конфигурации, объединяющие типы онтологии, промпты и примеры источников для конкретных областей. sage-wiki включает 8 встроенных пакетов, работающих оффлайн:

| Пакет | Аудитория | Ключевая онтология |
|-------|-----------|-------------------|
| `academic-research` | Исследователи | cites, contradicts, finding, hypothesis |
| `software-engineering` | Команды разработки | implements, depends_on, adr, runbook |
| `product-management` | Продуктовые менеджеры | addresses, prioritizes, user_story |
| `personal-knowledge` | Управление заметками | relates_to, inspired_by, fleeting_note |
| `study-group` | Студенты | explains, prerequisite_of, definition |
| `meeting-organizer` | Менеджеры | decided, assigned_to, action_item |
| `content-creation` | Авторы | references, revises, draft, published |
| `legal-compliance` | Юристы | regulates, supersedes, policy, control |

```bash
sage-wiki init --pack academic-research
sage-wiki pack install academic-research
sage-wiki pack apply academic-research --mode merge
sage-wiki pack list
```

Пакеты комбинируемы. Пакеты сообщества распространяются через реестр [sage-wiki-packs](https://github.com/xoai/sage-wiki-packs). См. [CONTRIBUTING.md](CONTRIBUTING.md).

## Внешние парсеры

sage-wiki включает встроенные парсеры для более чем 12 форматов. Для остальных форматов можно добавить внешний парсер в виде скрипта на любом языке. Протокол stdin/stdout.

```yaml
parsers:
  - extensions: [".rtf"]
    command: python3
    args: ["rtf_parser.py"]
    timeout: 30s
```

Безопасность: внешние парсеры выполняются с ограничением времени, очисткой переменных окружения и сетевой изоляцией на Linux. Требуется `parsers.external: true`. См. [CONTRIBUTING.md](CONTRIBUTING.md).

## Файлы навыков агента

sage-wiki имеет 17 MCP-инструментов, но агенты не будут их использовать, пока что-то в их контексте не укажет, *когда* обращаться к вики. Файлы навыков устраняют этот разрыв — сгенерированные фрагменты, которые обучают агентов, когда искать, что захватывать и как эффективно запрашивать.

```bash
# Генерация при инициализации проекта
sage-wiki init --skill claude-code

# Или добавление к существующему проекту
sage-wiki skill refresh --target claude-code

# Предварительный просмотр без записи
sage-wiki skill preview --target cursor
```

Это добавляет секцию поведенческих навыков в файл инструкций агента (CLAUDE.md, .cursorrules и т.д.) с триггерами, специфичными для проекта, рекомендациями по захвату и примерами запросов, полученными из вашего config.yaml.

**Поддерживаемые агенты:** `claude-code`, `cursor`, `windsurf`, `agents-md` (Antigravity/Codex), `gemini`, `generic`

Файл навыков предоставляет базовый универсальный шаблон — когда искать, что сохранять, как запрашивать — используя типы сущностей и связей из config.yaml. Для доменного поведения агента (триггеры исследований, шаблоны захвата встреч и т.д.) примените [пакет расширения](#пакеты-расширения):

```bash
sage-wiki init --skill claude-code --pack academic-research
```

Директория `skills/` пакета добавляет доменные триггеры вместе с базовым навыком. Запуск `skill refresh` перегенерирует только отмеченную секцию навыков — ваше остальное содержимое сохраняется.

## Захват знаний из разговоров с ИИ

sage-wiki работает как MCP-сервер, поэтому вы можете захватывать знания непосредственно из ваших разговоров с ИИ. Подключите его к Claude Code, ChatGPT, Cursor или любому MCP-клиенту — и просто попросите:

> «Сохрани то, что мы только что выяснили о пуле соединений, в мою вики»

> «Захвати ключевые решения из этой сессии отладки»

Инструмент `wiki_capture` извлекает элементы знаний (решения, открытия, исправления) из текста разговора с помощью вашего LLM, записывает их как файлы-источники и ставит в очередь на компиляцию. Шум (приветствия, повторные попытки, тупики) автоматически отфильтровывается.

Для отдельных фактов `wiki_learn` сохраняет запись напрямую. Для полных документов `wiki_add_source` загружает файл. Запустите `wiki_compile` для обработки всего в статьи.

Смотрите полное руководство по настройке: [Руководство по слою памяти агента](docs/guides/agent-memory-layer.md)

## Командная настройка

sage-wiki масштабируется от персональной вики одного человека до общей базы знаний для команд из 3-50 человек. Три модели развёртывания:

**Git-синхронизированный репозиторий** (3-10 человек) — вики живёт в Git-репозитории. Все клонируют, компилируют локально и пушат. Каталог скомпилированной `wiki/` отслеживается; база данных добавлена в `.gitignore` и пересобирается при каждой компиляции.

**Общий сервер** (5-30 человек) — запустите sage-wiki на сервере с веб-интерфейсом. Участники команды просматривают в браузере и подключают агентов через MCP по SSE.

**Федерация хабов** (мультипроектная) — каждый проект имеет свою вики. Система хабов объединяет их в единый интерфейс поиска через `sage-wiki hub search`.

```bash
# Хаб: регистрация и поиск по нескольким вики
sage-wiki hub add /projects/backend-wiki
sage-wiki hub add /projects/ml-wiki
sage-wiki hub search "deployment process"
```

**Что получают команды:**

- **Накопление институциональной памяти.** Что узнал один агент — знают все. Решения, соглашения и подводные камни, захваченные из любой сессии, доступны для поиска всем.
- **Выходные данные с контролем доверия.** [Система доверия к выходным данным](docs/guides/output-trust.md) помещает LLM-ответы в карантин до проверки обоснованности и подтверждения консенсусом. Галлюцинация одного агента не может отравить общий корпус.
- **Файлы навыков агента.** Сгенерированные инструкции обучают ИИ-агента каждого участника команды, когда обращаться к вики, что захватывать и как запрашивать. Поддерживаются Claude Code, Cursor, Windsurf, Codex и Gemini.
- **Авторизация по подписке для каждого пользователя.** Каждый разработчик использует свою подписку на LLM — никаких общих API-ключей в репозитории. В конфигурации указано `auth: subscription`; учётные данные хранятся индивидуально в `~/.sage-wiki/auth.json`.
- **Полный аудиторский след.** `auto_commit: true` создаёт git-коммит при каждой компиляции. Кто что изменил и когда.

```yaml
# Рекомендуемая конфигурация для команды
trust:
  include_outputs: verified    # карантин до верификации
compiler:
  default_tier: 1              # быстрая индексация, компиляция по запросу
  auto_commit: true            # аудиторский след
```

Смотрите [полное руководство по командной настройке](docs/guides/team-setup.md) для организации источников, рабочих процессов интеграции с агентами, конвейеров захвата знаний, вопросов масштабирования и готовых рецептов для стартапов, исследовательских лабораторий и команд с хранилищем Obsidian.

## Архитектура

![Архитектура Sage-Wiki](sage-wiki-architecture.png)

- **Хранилище:** SQLite с FTS5 (поиск BM25) + BLOB-векторы (косинусное сходство) + таблица compile_items для отслеживания уровня/состояния каждого источника
- **Онтология:** Типизированный граф сущностей и отношений с обходом BFS и обнаружением циклов
- **Поиск:** Улучшенный конвейер с индексацией фрагментов через FTS5 + вектор, расширением запросов через LLM, переранжированием через LLM, RRF-слиянием и 4-сигнальным расширением графа. Ответы поиска сигнализируют о нескомпилированных источниках для компиляции по запросу.
- **Компилятор:** Многоуровневый конвейер (Уровень 0: индекс, Уровень 1: эмбеддинг, Уровень 2: разбор кода, Уровень 3: полная LLM-компиляция) с адаптивным обратным давлением, параллельным извлечением Pass 2, кэшированием промптов, Batch API (Anthropic + OpenAI + Gemini), отслеживанием стоимости, компиляцией по запросу через MCP, оценкой качества и каскадным осознанием. Эмбеддинги включают повторные попытки с экспоненциальной задержкой, опциональное ограничение скорости и mean-pooling для длинных входных данных. 10 встроенных парсеров кода (Go через go/ast, 8 языков через regex, извлечение ключей из структурированных данных).
- **MCP:** 17 инструментов (6 чтение, 9 запись, 2 составных) через stdio или SSE, включая `wiki_compile_topic` для компиляции по запросу и `wiki_capture` для извлечения знаний
- **TUI:** bubbletea + glamour — 4-вкладочная терминальная панель (обзор, поиск, Q&A, компиляция) с отображением распределения по уровням
- **Веб-интерфейс:** Preact + Tailwind CSS, встроенный через `go:embed` с тегом сборки (`-tags webui`)
- **Scribe:** Расширяемый интерфейс для извлечения знаний из разговоров. Session scribe обрабатывает JSONL-транскрипты Claude Code.
- **Пакеты:** Система пакетов расширения — 8 встроенных пакетов, Git-реестр, жизненный цикл установки/применения/удаления/обновления, транзакционное применение с откатом через снимки.
- **Внешние парсеры:** Подключаемые парсеры форматов файлов через subprocess-протокол stdin/stdout. Песочница с таймаутом, очисткой окружения и сетевой изоляцией.

Без CGO. Чистый Go. Кроссплатформенный.
